# 35. 実務保守シミュレーション演習

## 目的

既存Strutsシステムの保守案件を想定し、調査、修正、テスト、レビュー、リリースまでを体験します。

## シナリオ

あなたは既存のStruts製会員管理システムの保守担当者です。

次の依頼が来ました。

```text
会員検索画面で、メールアドレスの一部を入力しても検索できるようにしてほしい。
また、検索結果にステータスを表示してほしい。
```

## 作業手順

1. 現行仕様を確認する
2. 該当URLを特定する
3. Actionを確認する
4. Serviceを確認する
5. RepositoryのSQLを確認する
6. JSPを確認する
7. 修正方針を作る
8. 実装する
9. テストする
10. コードレビューを受ける
11. リリース手順を作る

## 調査観点

| 観点 | 確認内容 |
|---|---|
| URL | どのActionが処理しているか |
| 入力 | 検索条件はどこで受け取るか |
| SQL | 検索条件にemailがあるか |
| 表示 | JSPにstatus列があるか |
| テスト | 既存テストがあるか |

## 修正例

- SearchFormにemailを追加する
- Repositoryのwhere句にemail条件を追加する
- JSPの一覧テーブルにstatus列を追加する
- Serviceテストを追加する
- 画面テスト項目を追加する

## 障害対応シミュレーション

### 障害

```text
本番リリース後、会員一覧画面で500エラーが発生した。
```

### 対応

1. エラーログを確認する
2. 直近リリース差分を確認する
3. 再現手順を確認する
4. 原因箇所を特定する
5. 暫定対応を決める
6. 恒久対応を実装する
7. 再発防止策を記録する

## 提出物

1. 調査メモ
2. 修正方針
3. 修正コード
4. テスト項目表
5. レビューコメント対応表
6. リリース手順書
7. 障害報告書

## 評価観点

- 影響範囲を確認できているか
- ActionだけでなくService/Repository/JSPまで追えているか
- SQL修正に安全性があるか
- テスト観点が十分か
- 障害時にログから原因を追えるか
